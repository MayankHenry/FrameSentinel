# FrameSentinel — Architecture Decisions Log

One line per decision + why. Keep this updated daily — this is your interview
cheat sheet later, not documentation for its own sake.

| Date | Decision | Why |
|------|----------|-----|
| Day 1 | `cv2.VideoCapture` for ingestion (not raw GStreamer pipeline yet) | Handles file/webcam/RTSP via the same API — no code change needed when real RTSP cameras replace test videos. Revisit if RTSP reconnect handling proves insufficient. |
| Day 1 | Sample via `.grab()` skip + `.retrieve()` on target frames, not decode-every-frame-then-drop | `.grab()` is far cheaper than full decode; matters once running multiple camera streams concurrently. |
| Day 1 | Target FPS configurable, decoupled from source FPS | Inference load is bounded by target FPS regardless of camera's native frame rate — this is the actual lever for scaling to more concurrent streams later. |
| Day 1 | `sampled_frames` per logged second doesn't equal target FPS | Ingestion runs at max decode speed for offline files (batch throughput), not throttled to real-time playback. Only live RTSP streams are naturally paced by the camera. No code change needed — just a logging nuance worth explaining if asked. |
| Day 2 | YOLOv8n (nano) as the default model, not s/m/l | Smallest/fastest variant — right default until an actual latency-vs-accuracy benchmark justifies a bigger model. Benchmark comparison is a Week 2/Day 14 task, not assumed now. |
| Day 2 | `Detector` class only exposes `DetectionResult` (dataclass), never raw Ultralytics objects | Keeps the rule engine (Day 4+) decoupled from the detection library. Swapping YOLOv8 for another detector later should be a config change, not a rewrite across the codebase. |
| Day 2 | Per-frame inference latency logged from Day 2, not added later | Same metric feeds Day 13's dropped-frame/latency observability story — cheaper to log it from the start than retrofit. |
| Day 3 | Used Ultralytics' built-in `model.track()` (ByteTrack) instead of a standalone ByteTrack implementation | Same underlying algorithm, far less integration surface area. Trade-off: less "I built the tracker from scratch" story, more "I understand tracker integration + its failure modes" story — the latter is more defensible under questioning. |
| Day 3 | Centroid defined as bbox bottom-center, not box center | Bottom-center approximates where a person is actually standing (feet position), which is what matters for zone-containment checks in Day 4 — box center would be roughly torso height, less accurate for "is this person inside the polygon." |
| Day 3 | `tracker_cfg` restricted to a whitelist (`bytetrack.yaml`/`botsort.yaml`), not an arbitrary path | If this pipeline is ever wrapped behind an API, an unrestricted config path is a file-read/config-injection vector. Cheap to lock down now, easy to forget later. |
| Day 3 | RTSP credentials masked in all logs (`mask_credentials`) | Camera URLs often carry `user:pass@host` — never want that landing in a log file or terminal history, even in a local dev script. |
| Day 3 | Per-frame tracking exceptions caught and skipped (return empty result), not raised | A single corrupt/flaky frame shouldn't kill a multi-hour camera session. Bad frames are logged as warnings; the pipeline keeps running. |
| Day 4 | Zone-intrusion alerts are edge-triggered (fire on outside→inside transition only), not fired every frame a track remains inside | A person standing in a zone for 10s at 8fps would otherwise generate ~80 identical alerts before any dedup logic even runs. Cheaper to not create the spam than to filter it out downstream. Day 5's dedup then handles the harder case: rapid re-entry/exit at a zone boundary. |
| Day 4 | `rule_engine.py` has zero dependency on cv2/ultralytics/track.py — pure Python + stdlib only | Makes Day 6's unit tests trivial (plain objects, no GPU/model mocking) and means swapping the detector/tracker library later never touches this file. `TrackSnapshot` is a deliberately separate, minimal class from track.py's `TrackedObject` to enforce this boundary. |
| Day 4 | Point-in-polygon implemented as plain ray-casting, not `cv2.pointPolygonTest` | Keeps the rule engine's only dependency being Python itself — see above. Same result, no OpenCV import needed. |
| Day 4 | Zone config format is a flat JSON array (zone_id, camera_id, name, rule_type, polygon), loaded from a file | "JSON is fine before DB exists" per the original plan — this becomes the `zones` Postgres table in Day 8 with no schema surprises, since the shape was already validated at load time. |
| Day 4 | Built a throwaway `zone_picker.py` (click-to-draw on first frame) instead of hand-typing pixel coordinates | Hand-guessing polygon coordinates from a video's resolution is slow and error-prone. This tool is explicitly NOT part of the production pipeline — Day 12's React drag-to-draw UI replaces it entirely. |
| Day 5 | Loitering "stationary" allows movement up to `movement_threshold_px` before resetting the dwell timer, not zero-tolerance | Real people sway, shift weight, and have detection-jitter even standing still. A zero-tolerance check would almost never fire in practice — the tolerance is what makes the rule usable, not just theoretically correct. |
| Day 5 | Leaving a loitering zone clears dwell state entirely (re-entry restarts the timer, doesn't resume it) | Simpler mental model and avoids a gameable edge case (step out for a second, step back in, keep an old dwell clock). If "cumulative time in zone across multiple visits" is ever needed, that's a different, explicitly separate rule — not a hidden behavior of this one. |
| Day 5 | Dedup implemented as an abstract `AlertDeduplicator` with `InMemoryDeduplicator` as the only concrete impl for now | Matches the plan's "in-memory dict for now, Redis-backed later" — RuleEngine only ever calls `should_emit()`, so a `RedisDeduplicator` can be swapped in during Day 7 without touching RuleEngine at all. |
| Day 5 | Every alert (both zone_intrusion and loitering) passes through the deduplicator, not just loitering | Zone-intrusion's edge-triggering alone doesn't protect against a flickering detection rapidly crossing a zone boundary (in/out/in/out within a couple seconds) — verified this produces 1 alert instead of 4 with a 5s cooldown, in the test suite. |
| Day 5 | `Zone.__post_init__` rejects `loiter_seconds` set on a `zone_intrusion` zone (not just "requires it for loitering") | A config mistake — pasting a loitering zone and forgetting to change `rule_type` back — would otherwise silently do nothing instead of failing loudly at load time. |
| Day 5 | Added `--append` mode to `zone_picker.py`, plus loitering support (`--rule-type loitering --loiter-seconds N`) | Real bug hit during testing: the tool originally always overwrote `--output` entirely, so drawing a zone_intrusion zone, then re-running to add a loitering zone, silently deleted the first one. `--append` merges into the existing file and auto-numbers new zone_ids to avoid collisions. |
| Day 5 | `Zone` now rejects self-intersecting polygons at construction time (`is_simple_polygon` check) | Second real bug hit during testing: clicking points for multiple distinct zones without pressing `n` between them (or re-clicking near the starting point to "close" a shape) merges unrelated points into one meaningless, self-intersecting polygon. Ray-casting still returns *an* answer for a self-intersecting shape, just not a predictable one — this was silently producing wrong alert behavior instead of failing loudly. Now it fails at load time with a message pointing at the actual cause. |
| Day 5 | `zone_picker.py` now guards against re-clicking near a polygon's start point, and validates (via the same `is_simple_polygon` check) before a shape can be finalized with `n`/`s`, instead of only catching bad polygons later at load time | The same self-intersection mistake happened a second time even after adding load-time validation and documenting the correct usage — the tool needed to make the mistake impossible to commit in the first place, not just detectable afterward. Fixing it at the point of input (ignore the bad click, block the bad save) is more reliable than fixing it via instructions the user has to remember. |
| Day 5 | `load_zones_from_json` reads config files with `utf-8-sig` instead of plain `utf-8` | Real bug hit during testing: PowerShell's `Out-File -Encoding utf8` writes a UTF-8 BOM prefix, which plain `utf-8` decoding rejects outright. `utf-8-sig` strips a BOM if present and is otherwise identical to `utf-8` — a safe default for a config file that may be hand-edited on Windows. |
| Day 5 | Added a `--debug-centroids` flag to `run_zone_demo.py` that prints each zone's bounding box at startup and every tracked centroid per frame | Needed this to actually diagnose a real "zero alerts" case — the zones were correctly-formed polygons that simply didn't overlap where the tracked person actually was in frame (drawn in the upper half of a much taller frame than expected). Comparing raw numbers side by side was faster and more conclusive than guessing from the display window alone. |
